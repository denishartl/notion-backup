## Project Notes

**notion-backup: Added parallelization for performance**

Implemented concurrent backup operations:

1. **New `concurrency.py` module** - `RateLimiter` class using simple timing with threading.Lock for thread-safe rate limiting at 2.5 req/sec (below Notion's 3/sec limit)

2. **`RateLimitedNotionClient`** in client.py - Wraps all API methods with rate limiting and automatic retry on HTTP 429. Uses `@retry_on_rate_limit()` decorator that respects Retry-After header.

3. **Parallel file downloads** in files.py - Uses ThreadPoolExecutor with 10 workers. File downloads are not rate-limited by Notion API so can run heavily parallel.

4. **Parallel page/database fetching** in __main__.py - Uses ThreadPoolExecutor with 3 workers. Each worker respects the shared RateLimiter. Uses threading.Lock for aggregating results.

Key design decisions:
- Block recursion within pages stays sequential (complexity not worth marginal gain)
- Markdown writing stays sequential (has mutable state for parent/child hierarchy)
- Used threading instead of asyncio (simpler, works with existing notion-client SDK)
- Hardcoded concurrency values (3 for API, 10 for files) rather than making configurable

Expected speedup: ~5x overall (limited by Notion's 3 req/sec rate limit for API calls, but file downloads get ~10x faster)
